Get Started with HesabPay

Integrate our session-based payment gateway with this step-by-step guide

How HesabPay Works - Session-Based Payments

1. Client Requests Payment

Your client-side application requests a payment session from your server with item details

2. Server Creates Session

Your server calls HesabPay API and receives a unique, secure payment URL in the response

3. Redirect to Payment

Customer is redirected to the HesabPay checkout page using the generated URL (supports pay-as-guest)

4. Handle Completion

Customer is redirected back to your site, and you can handle the response via redirects or webhooks

Get Your API Key

Obtain your API key from the Developer Profile section

Obtaining an API Key

To begin integrating HesabPay into your application, the first step is to obtain an API key. This key is essential as it authenticates and authorizes your application to interact with HesabPay's services.

Access API Key Management:
  1. Navigate to the Developer section at sidebar
  2. Click on the API Key tab
Generate API Key:

Once you access the API key section, you can generate a new API key. Remember, this key should be kept confidential and not exposed to the public.

Note: Make sure to append API-KEY in the Authorization header before your API key.

Security Best Practices:

  • • Keep your API key secure and never commit it to version control
  • • Store it in environment variables or secure configuration
  • • Never expose it in client-side code or public repositories
  • • Regenerate your key if you suspect it has been compromised

Authorization Header Format: Authorization: API-KEY your_api_key_here

Choose Integration Method

Select between Simple Payment or Multi-Vendor Payment

Create Payment Session

Once you have your API key, you can create a payment session. This involves sending a request to HesabPay's API endpoint with your API key and a list of items the user wishes to purchase.

API Endpoint Details

The Create Payment Session API is our main endpoint for processing transactions. This API creates a secure payment session and returns a unique payment URL that you can redirect your customers to.

Request Format:

Method: POST

Endpoint: https://api.hesab.com/api/v1/payment/create-session

Authorization: API-KEY your_api_key

API Response:
  • • A secure payment URL for customer redirect
Implementation Example
<?php
// Create payment session endpoint
$api_key = 'your_api_key_here';
$api_url = 'https://api.hesab.com';

$data = [
    'email' => 'customer@example.com', // optional
    'items' => [
        ['id' => 'item1', 'name' => 'Product 1', 'price' => 45],
        ['id' => 'item2', 'name' => 'Product 2', 'price' => 20]
    ],
    'redirect_success_url' => 'https://yoursite.com/success', // optional
    'redirect_failure_url' => 'https://yoursite.com/failure'   // optional
];

$ch = curl_init();
curl_setopt($ch, CURLOPT_URL, $api_url . '/api/v1/payment/create-session');
curl_setopt($ch, CURLOPT_POST, true);
curl_setopt($ch, CURLOPT_POSTFIELDS, json_encode($data));
curl_setopt($ch, CURLOPT_HTTPHEADER, [
    'Authorization: API-KEY ' . $api_key,
    'Content-Type: application/json'
]);
curl_setopt($ch, CURLOPT_RETURNTRANSFER, true);

$response = curl_exec($ch);
$result = json_decode($response, true);
curl_close($ch);

if ($result['success']) {
    // Redirect customer to payment URL
    header('Location: ' . $result['payment_url']);
    exit;
} else {
    echo 'Error: ' . $result['message'];
}

Next Step: Redirect your customer to the payment_url to complete their payment. They can pay as a registered user or as a guest.

Handle Post-Payment Process

Process payment confirmations, redirects, and customer experience

Checkout Page

The checkout page provides a secure environment for customers to complete their payments. It supports both registered users and pay-as-guest options.

Features:
  • • Secure payment processing
  • • Pay-as-guest option (no account required)
  • • Multiple payment methods
  • • Mobile-responsive design
Handling the Post-Payment Process
✅ Payment Success

After successful payment, users are redirected to a confirmation page as visual confirmation of transaction success. Additionally, a custom event named paymentSuccess is emitted.

Redirect URL: Your specified success URL with transaction details
❌ Payment Failed

If payment fails, users are redirected to your failure URL with error information.

Redirect URL: Your specified failure URL with error details

Important Note:

The paymentSuccess event is designed for cross-domain communication, making it crucial for integrators to understand how to securely capture and handle this information. For reliable payment confirmation, consider implementing webhooks (covered in step 6).

Frontend Event Handling (Optional)

Capture payment success events in your frontend application

Frontend Event Handling (Optional)

Optional Feature: If you want to capture payment events directly in your frontend application for enhanced user experience, you can listen for the paymentSuccess event.

Example Code for Capturing the Event
window.addEventListener('message', function(event) {
  if (event.origin !== 'http://expected-origin.com') {
    // Validate the event origin
    return;
  }
  if (event.data.type === 'paymentSuccess') {
    console.log('Payment successful:', event.data);
    // Handle the payment success, e.g., update UI, log to server
  }
});
Benefits of Frontend Event Handling:
  • • Immediate UI updates after payment completion
  • • Enhanced user experience with instant feedback
  • • Ability to trigger client-side actions
  • • Better integration with single-page applications

Important Security Note:

This event is for enhancing user experience and ensuring the integrity and continuity of the transaction flow. For security and functional effectiveness, it's vital to implement additional validation and error handling around the incoming data.

Note: This step is optional. You can skip this if you prefer to handle payment confirmation only through redirects and webhooks.

Multi-Vendor Setup (Optional)

Configure payment distribution for marketplace scenarios

Multi-Vendor Payment Distribution

Use Case: Perfect for marketplaces where items purchased by customers belong to different vendors. After a successful payment, you can automatically distribute the amounts to respective vendor accounts.

Example: Customer buys items worth $80 from 3 different vendors - automatically split payment between them.

API Workflow
1Customer completes payment through regular payment session
2You receive payment confirmation via webhook
3Call multi-vendor API to distribute payment to vendors
4Each vendor receives their portion automatically
API Request Example
// POST https://api.hesab.com/api/v1/payment/send-money-MultiVendor
{
  "pin": "aVhMuXVLTSeITdwBls7ludX8MGU=", // Your encrypted PIN
  "vendors": [
    {"account_number": "0794825911", "amount": 30},
    {"account_number": "0794825001", "amount": 25}, 
    {"account_number": "0742825911", "amount": 25}
  ]
}

Important Security Note

Your account PIN must be encrypted before sending. We provide encryption methods in your developer profile.

API Limitations

  • • Maximum 16 vendors per transaction
  • • No duplicate account numbers allowed
  • • Total amount must not exceed your balance
  • • All vendor accounts must be valid HesabPay accounts
